Security through obscurity

Dec 14, 2015
2 minute read

Security through obscurity - meaning that you assume that no one is going to figure out flaws in your system, it is just a matter of time that you could be targeted.

Keeping your door key under the welcome mat, or keeping your passwords under your desk.

I’ve had the telnet port opened on some random port number, this should keep my server safe? This assumption can go wrong completely. One can easily identify the service using fingerprinting techniques.

So, what are we talking here? Anything that is hidden to be disclosed is the approach to implement security through obscurity. Suppose I am using secure shell for various administrative operations. What would be the fundamental thing to do so that the access is secured. Obviously, I can achieve 100% security by not using it at all, but that is not possible. So, on practical grounds I would use key or public key based authentication. Similarly I could have things like enforcing strong passphrase, disabling all users unless otherwise specified, disable remote connections etc. So, I could implement these and many other practices to strengthen the security involved in secure shell access. However, in contrast to that if I prefer not to carry all of those steps and just change the SSH port from 22 to say port 2345, thinking that I would be able to hide the service, and thus protect it, I would be trying to implement security through obscurity. Thus, be an idiot. Because, it wouldn’t take more then few minutes to identify the obscured port specially when the service is exposed on a public network.

The standard security approach thus suggests one to follow the standards and well accepted practices to enhance security and not to hide the loop holes that you might have. However, if we think in a different most of our security practices rely on obscurity. Use of passwords, pass-phrases or high bit encryption keys are all ways of obscuring security in a different way. All of our adapted security practices implement security through obscurity in one way or the other. The only important part being the ability to determine the exact breakeven point. Failing to identify the exact point would either increase the security risk or decrease the usability.
